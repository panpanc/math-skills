# Chapter 4: Computing Homology in Practice

## Learning objectives
- Build boundary matrices from oriented simplicial data.
- Compute kernels, images, and quotient groups in concrete examples.
- Use sanity checks to catch sign and indexing errors.

## Prerequisite chapters
- Chapter 1
- Chapter 2
- Chapter 3

## Chapter target concept/theorem
Finite simplicial homology is computable from integer boundary matrices.

## 1. Destination and Why It Matters

Chapter 3 gave the formula $H_n=\ker\partial_n/\operatorname{im}\partial_{n+1}$. This chapter makes that formula executable. We need a deterministic pipeline that works on complexes larger than toy drawings.

The destination is a repeatable workflow: orient, index, build matrices, compute kernel/image, form quotient, validate.

Callback: we now operationalize the chain and quotient ideas from Chapters 2 and 3.

## 2. Foundation Layer

### Primitive 1: Stable simplex indexing
- What it is: A fixed ordering of simplices in each degree.
- Why it is load-bearing: Matrix rows/columns have meaning only with stable indices.
- Tiny numeric example: Edge basis $(e_{01},e_{02},e_{12},e_{13})$ fixes row order in $\partial_2$.
- Failure mode: Reordering basis midstream invalidates computed kernels.

### Primitive 2: Signed incidence entries
- What it is: Entry is $+1$, $-1$, or $0$ depending on induced orientation.
- Why it is load-bearing: Unsigned entries erase cancellation.
- Tiny numeric example: Shared edge can contribute $+1$ in one triangle and $-1$ in adjacent triangle.
- Failure mode: Absolute-value incidence gives wrong homology.

### Primitive 3: Integer kernel computation
- What it is: Solve $\partial_nx=0$ in integer coordinates.
- Why it is load-bearing: Chains are integer objects.
- Tiny numeric example: For cycle matrix of a triangle graph, $(1,1,1)^T$ is in kernel.
- Failure mode: Solving only over reals can hide integer structure.

### Primitive 4: Image as column span
- What it is: $\operatorname{im}\partial_{n+1}$ is generated by columns of $\partial_{n+1}$.
- Why it is load-bearing: Boundaries are exactly these generated vectors.
- Tiny numeric example: Column $(1,-1,1)^T$ generates a rank-1 boundary subgroup.
- Failure mode: Confusing row span and column span.

### Primitive 5: Betti rank check
- What it is: Free-rank check $\beta_n=\dim\ker\partial_n-\dim\operatorname{im}\partial_{n+1}$ over field or free-part intuition.
- Why it is load-bearing: Fast consistency check for computations.
- Tiny numeric example: Circle has $\beta_0=1$ and $\beta_1=1$.
- Failure mode: Treating Betti ranks as full group structure when torsion appears.

### Primitive 6: Euler-characteristic sanity check
- What it is: Alternating simplex counts and alternating Betti counts should match free-rank behavior.
- Why it is load-bearing: Catches indexing or sign bugs.
- Tiny numeric example: Hollow triangle has $3-3+0=0$ and $\beta_0-\beta_1=1-1=0$.
- Failure mode: Using this check as a complete proof of correctness.

## 3. Bottom-Up Thinking Stages

### Stage 1: Freeze orientations and bases
**Motivation**:
Bottleneck: Computation is meaningless if basis semantics drift.
Why-now trigger: Same complex gave conflicting outputs under different hidden orderings.
Rejected shortcut: Choose basis order ad hoc for each map.
Stage objective: Lock orientation and basis order once at start.
Payoff signal: Every matrix entry has unambiguous interpretation.

**Intuition**:
Mental model: Fix coordinate axes before doing linear algebra.
Where it fails: Fixed axes alone do not prevent sign errors in incidence generation.

**Construction step**:
Uses: Primitive 1 and Chapter 1 orientation.
Choose ordered bases $\mathcal{B}_0,\mathcal{B}_1,\mathcal{B}_2$ and write them explicitly before matrix assembly.

**Worked example**:
Example 1
Setup: Triangle graph with edges $(e_{01},e_{12},e_{20})$ and vertices $(v_0,v_1,v_2)$.
Compute: Fix these orders globally.
Interpret: Later kernel vector $(1,1,1)^T$ has stable meaning.
Common mistake: Swapping edge order later but reusing old vector coordinates.

Example 2
Setup: Two-triangle patch sharing one edge.
Compute: Orient triangles consistently so shared edge is opposite induced orientation.
Interpret: Cancellation setup is baked in.
Common mistake: Orienting both faces with same shared-edge direction.

**What this unlocks next**:
Consistent boundary matrix construction.
Remaining gap: Need explicit matrix assembly rule.

### Stage 2: Build boundary matrices
**Motivation**:
Bottleneck: Symbolic boundaries do not scale to many simplices.
Why-now trigger: We need algorithmic input for kernel/image routines.
Rejected shortcut: Expand formulas manually for each query.
Stage objective: Assemble $[\partial_n]$ column by column.
Payoff signal: Boundary operators become sparse integer matrices.

**Intuition**:
Mental model: Each column is one simplex boundary in coordinates.
Where it fails: Sparse structure can hide geometry unless basis labels are tracked.

**Construction step**:
Uses: Primitive 2 and Chapter 2 formula.
For simplex $\sigma_j^n$, compute $\partial_n\sigma_j^n$ and write coefficients in rows of $(n-1)$-simplex basis.

**Worked example**:
Example 1
Setup: Hollow triangle.
Compute: $[\partial_1]=\begin{bmatrix}-1&0&1\\1&-1&0\\0&1&-1\end{bmatrix}$ and $[\partial_2]=0$.
Interpret: Matrix already suggests one cycle and one component.
Common mistake: Using unsigned entries.

Example 2
Setup: Filled triangle.
Compute: $[\partial_2]=\begin{bmatrix}1\\-1\\1\end{bmatrix}$ with chosen edge order.
Interpret: One boundary relation appears in degree 1.
Common mistake: Changing row ordering without recomputing signs.

**What this unlocks next**:
Kernel and image computations are now concrete.
Remaining gap: Need integer solutions for kernels.

### Stage 3: Compute cycle groups as kernels
**Motivation**:
Bottleneck: Matrix assembly alone does not produce homology.
Why-now trigger: We need generators of $Z_n$.
Rejected shortcut: Guess cycle generators from picture only.
Stage objective: Solve $\partial_nx=0$ in integer coordinates.
Payoff signal: A basis/generator set for $Z_n$ is explicit.

**Intuition**:
Mental model: Kernel vectors are balanced flows with zero boundary leakage.
Where it fails: Different kernel bases represent same subgroup and need interpretation care.

**Construction step**:
Uses: Primitive 3 and Stage 2 matrices.
Solve linear system $[\partial_n]x=0$ using row operations or Smith-style integer reduction.

**Worked example**:
Example 1
Setup: Hollow triangle $\partial_1$ matrix.
Compute: Solution space is $x=t(1,1,1)^T$, so $Z_1\cong\mathbb{Z}$.
Interpret: Exactly one independent 1-cycle direction.
Common mistake: Concluding kernel is trivial because matrix has rank $2$.

Example 2
Setup: Interval with one edge.
Compute: $\partial_1=[-1\;1]^T$ gives kernel $0$ in degree 1.
Interpret: No nontrivial 1-cycle.
Common mistake: Confusing kernel dimension with codomain size.

**What this unlocks next**:
Need image subgroup to remove trivial cycles.
Remaining gap: Compute boundaries as column spans.

### Stage 4: Compute boundary subgroups as images
**Motivation**:
Bottleneck: Kernel alone still overcounts classes.
Why-now trigger: Need $B_n$ to form quotient.
Rejected shortcut: Assume image rank from geometric intuition.
Stage objective: Determine $\operatorname{im}\partial_{n+1}$ from columns.
Payoff signal: Trivial-cycle subgroup is explicit.

**Intuition**:
Mental model: Columns of $\partial_{n+1}$ are primitive boundary generators.
Where it fails: Over reals, same span can hide different integer lattice structure.

**Construction step**:
Uses: Primitive 4.
Compute column span of $[\partial_{n+1}]$ in chosen $C_n$ coordinates.

**Worked example**:
Example 1
Setup: Filled triangle with $[\partial_2]=(1,-1,1)^T$.
Compute: $B_1=\langle(1,-1,1)^T\rangle$.
Interpret: One generated boundary direction in $C_1$.
Common mistake: Claiming $B_1\cong\mathbb{Z}^3$ from vector length.

Example 2
Setup: Hollow triangle with $\partial_2=0$.
Compute: $B_1=0$.
Interpret: No 2-chain fillers exist.
Common mistake: Importing boundary relation from filled case.

**What this unlocks next**:
Now quotient $Z_n/B_n$ can be computed.
Remaining gap: Need quotient simplification and interpretation.

### Stage 5: Form quotient groups and report results
**Motivation**:
Bottleneck: Separate kernel/image data is not yet the invariant.
Why-now trigger: We must identify cycles modulo boundaries.
Rejected shortcut: Subtract dimensions and stop.
Stage objective: Compute full group structure when possible.
Payoff signal: Output as $0$, $\mathbb{Z}^r$, or with torsion factors.

**Intuition**:
Mental model: Quotient removes directions explainable by higher-dimensional fillings.
Where it fails: Rank-only summary misses finite-order classes.

**Construction step**:
Uses: Chapter 3 quotient logic and Stage 3/4 outputs.
Compute $$H_n=Z_n/B_n$$ using generator-relation simplification or Smith normal form (integer diagonal reduction that exposes free and torsion factors).

**Worked example**:
Example 1
Setup: Hollow triangle.
Compute: $Z_1\cong\mathbb{Z}$ and $B_1=0$, so $H_1\cong\mathbb{Z}$.
Interpret: One essential loop class.
Common mistake: Reporting only $\beta_1=1$ and omitting group structure.

Example 2
Setup: Filled triangle.
Compute: $Z_1\cong\mathbb{Z}$ and $B_1\cong\mathbb{Z}$ with same generator, so $H_1=0$.
Interpret: Loop collapses as boundary.
Common mistake: Assuming nonzero cycle generator always survives.

**What this unlocks next**:
Ready for larger examples with confidence.
Remaining gap: Need robust validation checks against mistakes.

### Stage 6: Apply sanity checks systematically
**Motivation**:
Bottleneck: Small sign/index errors can flip results dramatically.
Why-now trigger: Hand and code pipelines both need deterministic checks.
Rejected shortcut: Trust first output from reduction routine.
Stage objective: Validate with $\partial_{n-1}\partial_n=0$ and Euler checks.
Payoff signal: Inconsistencies are caught before interpretation.

**Intuition**:
Mental model: Invariants come with built-in test cases.
Where it fails: Passing sanity checks does not prove no torsion mistakes.

**Construction step**:
Uses: Primitive 5 and Primitive 6.
Check matrix product $[\partial_{n-1}][\partial_n]=0$. Compare alternating simplex counts and Betti free-rank expectations.

**Worked example**:
Example 1
Setup: Hollow triangle matrices.
Compute: Product $[\partial_0][\partial_1]=0$ and $\chi=0$ matches $\beta_0-\beta_1=0$.
Interpret: Pipeline consistency confirmed.
Common mistake: Using Euler match as final proof of full group decomposition.

Example 2
Setup: Two disconnected edges.
Compute: $\beta_0=2$, $\beta_1=0$, and simplex count gives $\chi=2$.
Interpret: Component count aligns with computation.
Common mistake: Misreading close drawing as connected graph.

**What this unlocks next**:
Reliable computation for moderate-size complexes.
Remaining gap: Absolute groups still do not express subcomplex comparison.

### Stage 7: Motivate relative computations
**Motivation**:
Bottleneck: Absolute homology alone cannot encode what changes relative to a subcomplex.
Why-now trigger: Applications require pairwise comparison $(K,A)$.
Rejected shortcut: Subtract absolute Betti numbers.
Stage objective: Frame need for relative chain complexes.
Payoff signal: Clear conceptual handoff to Chapter 5.

**Intuition**:
Mental model: Absolute homology is a snapshot; relative homology is a constrained comparison.
Where it fails: Snapshot metaphor hides exact map relationships.

**Construction step**:
Uses: Chapter 3 invariant and Stage 6 diagnostics.
Observe that $H_n(K)$ and $H_n(A)$ alone do not determine how classes map between spaces; this motivates quotient chains $C_n(K)/C_n(A)$.

**Worked example**:
Example 1
Setup: Disk $K$ and boundary circle $A$.
Compute: $H_1(K)=0$ but $H_1(A)\cong\mathbb{Z}$.
Interpret: Need structural relation, not arithmetic difference.
Common mistake: Writing $H_1(K,A)=H_1(K)-H_1(A)$.

Example 2
Setup: Interval and its endpoints.
Compute: Absolute groups miss the constrained path class captured relative to endpoints.
Interpret: Relative viewpoint adds information.
Common mistake: Treating relative theory as optional abstraction.

**What this unlocks next**:
Relative homology and exact sequences.
Remaining gap: Need formal pair-level definitions and maps.

## 4. Final Concept/Theorem Statement

Formal statement: For a finite simplicial complex, once bases are fixed and boundary matrices are built, homology is computable by finite integer linear algebra:
$$
H_n(K)=\ker[\partial_n]/\operatorname{im}[\partial_{n+1}].
$$

Near-miss/non-example: Computing only real-valued ranks and treating them as full group answers fails when torsion is present.

Wrong mental model: Homology computation is just visual loop counting.

Correction: It is a structured algebraic pipeline requiring orientation control, signed incidence matrices, and quotient-group simplification.

Scope boundary: This chapter treats absolute computation workflows. It does not yet introduce relative chains, long exact sequences, or functorial invariance proofs.

## 5. How the Thinking Process Could Have Invented It

Chapter 3 provided the correct formula but not a scalable procedure. The invention path was to represent boundaries as matrices, then reuse linear algebra for kernels and images, and finally apply quotient logic mechanically.

Failed attempt 1: Compute homology by visual inspection. This fails for dense complexes and interacting cycles.

Failed attempt 2: Use only rank subtraction over reals. This fails to capture full integer-group structure and can hide subtle errors.

Design rule: Separate representation discipline (orientation/indexing) from invariant extraction (kernel-image quotient), then enforce sanity checks after each run.

Bridge to next chapter: In Chapter 5 we build relative homology and the long exact sequence to compare a complex against a chosen subcomplex.
